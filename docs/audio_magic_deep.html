The premise: store digital data — eventually a small language model — on an ordinary audio cassette, by turning bytes into sound, recording that sound to tape, and reading it back. Tape is a hostile analog channel. Everything below is the engineering of squeezing the most reliable bits per second through it.

0 · The signal chain at a glance

Every scheme in this repo is the same pipeline. Bytes in, audio out, tape in the middle, bytes out — ideally identical bytes.

payload bytes │ ① frame: MAGIC "CA" │ len(2) │ payload │ CRC32(4) (integrity) │ ② outer code (analytic): RS / fountain → net = gross × code_rate │ ③ bits → symbols (the "modulation" — where the magic lives) ▼ [ chirp preamble 800→3200 Hz, 0.25 s ] + [ data symbols ] = float32 @ 48 kHz │ ▼ ───────── THE CHANNEL ───────── tape record → wow/flutter → band-limit → tape hiss (AWGN) → dropouts → playback → (acoustic hop: speaker → mic) OR (electrical: line-out → ADC) ▼ ④ coarse sync: cross-correlate against the chirp → find data start ⑤ global speed recovery: chirp-pair spacing → resample to nominal 48 kHz ⑥ demodulate symbols → bits (FFT energy / matched filter / OFDM equalize) ▼ recovered bytes → CRC check → byte-exact? (and: raw BER for the projection)

The interesting choices are at ③ (how a symbol encodes bits) and ⑥ (how you recover it from a mangled waveform). Steps ①, ④, ⑤ are shared scaffolding. Step ② is where the reliability lives and is mostly analytic in this work (see §2).

1 · The channel — what a cassette does to sound

You cannot design a modem without a faithful model of the enemy. src/channel.py applies five impairments, in order. The "normal" consumer-deck preset is the one every campaign number is measured on.

1.1 Band-limiting

A 5th-order Butterworth low-pass (zero-phase, sosfiltfilt). Tape head-gap geometry and bias rolloff kill the top octave. Usable band ≈ 400 Hz – 10.5 kHz; the cutoff per preset is 9–15 kHz.

1.2 Wow & flutter — the time axis wobbles

The defining tape impairment. The capstan/pinch-roller never turns at exactly constant speed, so the playback time base warps. Modeled as a slow instantaneous-rate perturbation built from two sinusoids (wow ≈ 0.55 Hz, flutter ≈ 4.8 Hz) plus noise, normalized to a target WRMS, then integrated into a warped time vector and resampled:

inst(t) = a₁·sin(2π·0.55·t+φ₁) + a₂·sin(2π·4.8·t+φ₂) + 0.18·n(t)
inst ← inst / RMS(inst) × wf_wrms # a₁~N(1,.08), a₂~N(.45,.05)
warped_t = t + cumsum(inst)/fs
y = interp(t, warped_t, y) # resample onto the wobbling clock

This is why orthogonality and coherence are fragile: a tone that was an integer number of cycles in a symbol window no longer is, after the window is stretched. Non-coherent (energy-only) detectors tolerate it; coherent (phase-tracking) ones suffer. Normal preset = 0.10% WRMS; the real acoustic loop measured 2.2% (§12).

1.3 Additive noise — tape hiss

White Gaussian noise scaled to the preset SNR over the signal power:

noise_power = mean(y²) / 10^(SNRdB/10); y ← y + 𝒩(0, √noise_power)

Normal = 42 dB. This is what caps high-order constellations: QAM needs SNR-per-symbol that hiss steals.

1.4 Dropouts — burst erasures

Oxide flaws / dust briefly drop the level. Modeled as a Poisson number of Tukey-faded attenuation bursts (rate 0.30/s, length 6 ms on normal), each knocking the level down to 0–8%:

n_bursts ~ Poisson(rate·duration); per burst: y[s:e] ×= att + (1−att)·(1−tukey)

Bursts are correlated errors — they hit a contiguous run of symbols. This is precisely what killed soft-FEC (§7): short-constraint codes can't span a burst. The cure is interleaving + erasure coding (§2).

1.5 Steady speed offset & soft-clip

A constant deck-speed mismatch is a whole-signal resample (ratio 1+δ); the real loop ran 0.88× (an iPhone-Continuity capture artifact). Optional tanh soft-clip models record-level overdrive — push the level too hot and intermodulation products (the −10.5 dB IMD floor) bloom and smear dense carriers.

2 · The harness & the net-rate projection

Every scheme is judged by one apples-to-apples pipeline in src/hyp_common.py, so comparisons are fair. A scheme is just three things: modulate(bits)→audio, demodulate(audio)→bits, and a declared gross_bps. No oracle timing — sync must come from the signal.

2.1 The two metrics

  • gross_bps — information bits per second of emitted audio, including the preamble/framing overhead. The raw rate the channel runs at.
  • raw BER — post-sync, pre-FEC bit-error rate, Monte-Carlo'd over many seeds through the channel.

2.2 From BER to a code rate

The key idea: a given raw BER permits some maximum outer-code rate R that still recovers a megabyte-scale file whole. project_to_cassette() reads R from a deliberately conservative step table (below Shannon's hard-decision limit R = 1 − H₂(p), so we never over-claim):

raw BER ≤1e-61e-51e-41e-33e-31e-22e-24e-26e-28e-21e-1>
code rate R.98.95.92.85.80.70.55.40.30.20.10~0
net_bps = gross_bps × R
MB/cassette = net_bps × seconds × tracks / 8 / 10⁶

C90 usable = 5374 s, C60 = 3574 s; stereo = 2 independent tracks. So 1076 net bps → 1076×5374×2/8/10⁶ = 1.445 MB on a C90.

This is the lever the whole campaign exploits. Notice the table is a staircase. MFSK-32 sits at BER 2.0e-3 → just above the 1e-3 knee → pinned at R = 0.80. Pushing BER below 1e-3 jumps R to 0.85 (+6% net for free); below 1e-4 jumps to 0.92 (+15%). Several hypotheses win purely by stepping down one stair.

2.3 Erasures are cheaper than errors

If a scheme knows a slot was lost (a CRC'd frame failed, a subcarrier is a known null), that's an erasure, not an error. An ideal MDS/fountain code (RaptorQ) recovers k source symbols from k/(1−e) received ones — overhead e/(1−e), far cheaper than coding blind against the same loss as bit-errors. P_full (whole-file recovery probability) must stay 1.0 for any result to count as "no increase in error rate."

3 · Baseline — BFSK / CAS3 (B0)

The shipping codec: binary FSK (two tones, 1 bit each), framed into 256-byte CRC'd frames with a 2 s leader + chirp. Robust front-end does chirp-correlation sync, a brute-force tape-speed search (0.94–1.06×), DC/gain normalization. Measured: gross 538 bps, in-frame BER ≈ 1e-6, net ≈ 478 bps, 0.64 MB / C90-stereo. Rock-solid, but slow — one bit per tone per symbol is the floor everything tries to beat.

4 · The frontier — MFSK-32 (Olivia lineage)

Instead of 2 tones, use M = 32 orthogonal tones; each symbol lights exactly one, carrying log₂32 = 5 bits. Detection is non-coherent FFT energy: take the FFT of the symbol window, read the 32 tone bins, pick the largest. No phase tracking → flutter-tolerant.

4.1 The orthogonality condition

Tones are spaced Δf apart across the usable band; the symbol period is set so each tone is an integer number of cycles in the window (so the FFT bins don't leak into each other):

Δf = bandwidth / (M−1) # 32 tones over ~400–10000 Hz → Δf ≈ 309 Hz
Tsym = 1 / Δf # ≈ 3.23 ms → bins are exactly orthogonal
gross_bps = log₂(M) / Tsym = log₂(M)·Δf = log₂(M)·BW/(M−1)

4.2 Why it wins

Non-coherent energy detection has a low BER on this channel (2.0e-3), and the projection rewards low BER. Result: gross 1412, net 1076 bps, 1.45 MB / C90-stereo, 2.25× the BFSK baseline, P_full = 1.0. This is the number every capacity hypothesis is measured against.

The tension that drives everything: more bits/symbol means either more tones (narrower Δf → more flutter-sensitive, lower SNR per tone) or higher-order constellations (more SNR-hungry). MFSK is robust but spends the entire band to carry only 5 bits/symbol. That inefficiency is the opening.

5 · C1 — Gray coding + band re-tune ACCEPT 1.12×

Mechanism. When MFSK mistakes a tone for its neighbor (the common error), a binary symbol→bit map can flip several bits at once. A Gray code orders the map so adjacent tones differ by exactly one bit — an adjacent-tone slip costs 1 bit instead of ~½·log₂M. That cuts raw BER 2–3×.

gray(i) = i XOR (i >> 1) # binary→Gray; demod inverts the permutation

Result. Gray alone cut M=32 BER 2.0e-3 → 1.6e-3 — but both sit in the same 0.80 stair, so net was flat. The actual win came from operating-point tuning: M=16 over a narrower 400–7000 Hz band gives a wider Δf = 440 Hz (more flutter margin) and a 12% higher gross (1587 bps) while holding BER in the 0.80 bucket. Net 1209 bps, 1.62 MB. A modest, honest win; Gray is a real but small lever here because the staircase swallowed it.

6 · C2 — Combinatorial k-of-M multitone FSK ACCEPT 2.24×

Mechanism — the strongest structural idea. Instead of lighting one of M tones, light K of M simultaneously. The number of distinct K-subsets is C(M,K), so each symbol carries:

bits/symbol = ⌊log₂ C(M,K)⌋

For M=48, K=6: C(48,6) = 12 271 512 → log₂ = 23.55 → 23 bits/symbol versus 5 for MFSK-32, on the same tone grid and same non-coherent demod (pick the top-K energy bins).

6.1 The combinatorial number system

You need a clean bijection between an integer in [0, C(M,K)) and a specific K-subset, both ways. The combinatorial number system does exactly this: every integer has a unique representation as a sum of binomials, which is a K-subset. Encode = integer→subset→light those bins; decode = top-K bins→subset→integer→bits. Placing all M tones at exact FFT bin centers makes top-K detection lossless on a clean signal (sanity BER = 0).

6.2 The tradeoff and the result

Splitting symbol energy across K tones costs ≈ 10·log₁₀(K) dB of per-tone SNR (~7.8 dB at K=6) → raw BER rises to 5.6e-3. But that's still inside the 1e-2 stair (R = 0.70), and the gross explosion (3618 vs 1412) crushes the penalty. Net 2412 bps, 3.24 MB / C90-stereo, 2.24×. M=16 configs collapse (Δf too wide × flutter → BER > 4e-2). The sweet spot is many narrow tones with enough K to harvest the combinatorial gain.

7 · C3 — Soft-decision FEC credit REJECT (fair)

Hypothesis. The projection table is hard-decision (it only knows the final BER). But MFSK FFT bin energies are natively soft — the gap between the top-1 and top-2 bins is a reliability (an LLR). A soft-decision decoder buys ~1.5–2 dB of coding gain, i.e. recovers at a higher code rate than the hard table grants for the same BER → more net at equal physical error.

What was actually built. A rate-3/4 punctured K=7 convolutional code (generators g₁ = 0o133, g₂ = 0o171) with a soft Viterbi decoder fed log-sum-exp LLRs from the FFT bins. Swept the rate up until post-FEC whole-file error just held zero across seeds.

Why it failed (honestly). It reached P_full = 1.0 only at R = 3/4 → net 1081 vs MFSK-32's 1076: a 0.5% gain, far below the 1.15× bar. R = 5/6 catastrophically failed on a single seed (post-FEC BER 0.324). The culprit is the channel, not a bug: cassette burst dropouts cluster inside the short constraint window, and the FEC's extra symbols (more audio) cost more than the soft gain buys. Lesson: soft decoding helps against hiss (AWGN), not against bursts. The fix would be a deep interleaver + a long-constraint/LDPC code — listed as future work.

8 · C4 — Bit-loaded OFDM-QAM (water-filling) ACCEPT 3.69× — new frontier

OFDM divides the band into many narrow subcarriers carried in parallel by an inverse FFT, each holding a QAM symbol. A cyclic prefix (copy the FFT tail to the front) absorbs timing slop and short echoes. Bit-loading / water-filling is the trick: don't load every subcarrier equally — measure each one's SNR and assign its QAM order accordingly (strong → QPSK/16-QAM, weak → BPSK, hopeless → 0 bits). Pilots (known symbols sprinkled in frequency) let the receiver estimate and equalize each subcarrier's gain/phase per symbol.

8.1 The diagnosis that unlocked it

A prior OFDM attempt in this repo (H3) was rejected as "flat loading wastes spectral efficiency." That was wrong. The real cause, found here, was a timing bug: a long FFT + wow/flutter clock drift + a seed-dependent ~25–40-sample sync offset walked the fixed symbol windows off-boundary, collapsing whole frames to ~50% BER (−20 dB EVM). The spectral efficiency was never the problem.

8.2 The three fixes

  1. Short FFT, N = 256 (~5.3 ms symbol) + CP = 32 → intra-symbol flutter ICI stays small and uniform.
  2. Wide acquisition + per-symbol tracking: a ±50-sample search on symbol 0 undoes the coarse-sync offset, then incremental ±4 tracking with a frequency-domain residual-slope correction from the embedded pilots holds EVM at ~18 dB across the whole frame.
  3. Channel-probe bit-loading: push known QPSK through the channel on calibration seeds (disjoint from eval seeds — no leakage), measure per-subcarrier SNR (~13 dB median), then a gap-approximation bi = log₂(1 + SNRi/Γ) loads QPSK on the cleanest carriers, BPSK elsewhere (39 data SC + 13 pilots → 49 bits/OFDM-symbol).

Result. gross 4902, raw BER 4.7e-4 (below the 1e-3 knee → R = 0.85), net 3968 bps, 5.33 MB / C90-stereo, 3.69× — and at a lower BER than the frontier. Independently reproduced from scratch. Erasure-marking burst symbols was tested and hurt (net 3628 vs 3968) — the burst symbols still decode, so paying fountain overhead for them is wasteful.

Caveat that becomes central in §12: ~13 dB clean SNR is power-limited — it only supports QPSK, not the 16/64-QAM the framework allows (only ~12% of the band's Shannon bound). OFDM's gross is bounded by SNR, and that SNR assumption is exactly what a real tape may not honor.

9 · C5 — Faster-than-Nyquist tone packing REJECT (fair)

Mechanism (moonshot). Orthogonality says symbols must be spaced Tsym apart. Faster-than-Nyquist deliberately violates this: hop = α·Nburst with α < 1, overlapping consecutive Hanning-tapered tone bursts, then separating them with a coherent matched-filter bank (cos/sin projection, max-energy tone). More symbols per second through the same band — if the decoder can undo the controlled inter-symbol interference.

Result. It genuinely works: α = 0.78 packs symbols ~28% faster, lifting gross 1481 → 1846 bps, and the matched filter still separates tones (sanity BER 0). Net 1406 bps, 1.31× — a real gain, but below the 1.5× moonshot bar. Packing tighter (α ≤ 0.76) pushes ICI across the 1e-2 knee and the gross gain is more than eaten. The 42 dB SNR + flutter floor sets a hard ICI ceiling. "No improvement past 1.5×" was the pre-registered acceptable null, and the α-sweep documents exactly where it tops out.

Code: experiments/capacity/c5_ftn.py

10 · Sync, speed recovery, and the channel sounder

10.1 Chirp correlation — finding the data with no clock

Every burst starts with a linear up-chirp (800 → 3200 Hz, 0.25 s). The receiver cross-correlates the recording against a local copy; the correlation peak is the data start. A chirp is ideal: its autocorrelation is a sharp spike, it survives band-limiting, and it's immune to where the silence starts.

corr = correlate(audio, chirp, "valid"); data_start = argmax|corr| + len(chirp)

10.2 Global speed recovery — beating the 0.88× deck

The tape test brackets the whole 16-minute master with two chirps. The measured spacing between them vs. the manifest's expected spacing is the deck speed ratio. Resample the entire recording by that ratio and the nominal 48 kHz clock is restored before any per-section decode. In validation this recovered the simulated 0.88× clock to 0.00% error.

speed = measured_spacing / expected_spacing; audio ← resample(audio, 1/speed)

10.3 The sounder — measuring the real channel, fresh, every recording

~45 s at the front of the master: two 64-tone Schroeder-phased multitones (low PAPR → measure H(f) and SNR(f) per tone without clipping), a 12 s steady 3 kHz tone (zero-crossing/IF wobble → flutter WRMS and a second speed estimate), and 3 s of silence (noise floor). This means the analyzer never trusts a stale channel model — it characterizes this tape, this playback.

11 · The tape test v2 — a robustness ladder

The campaign numbers are simulation. experiments/tape_v2/ turns them into a physical experiment: one self-describing 16.65-min master2.wav, 9 modulation configs × ~88 reps each (794 sections), ordered robust → aggressive so a deck that fades partway still captures the easy rungs cleanly.

11.1 The frozen modem interface

Every config implements the identical contract so the assembler and analyzer treat them uniformly:

modulate(payload, config) → float32 @48k [chirp preamble + framed symbols]
demodulate(audio, config) → bytes | None # CRC-gated
tx_bits(payload, config) → ground-truth bit stream # for BER
rx_bits(audio, config) → best-effort recovered bits # for BER

Frame: "CA" │ len(2) │ payload │ CRC32(4). Sections are short (~96-byte payloads) so intra-section drift stays small — that's what lets them survive a real deck.

11.2 The crucial measurement fix — BER, not just pass/fail

The first build scored only per-frame pass/fail (CRC byte-exact). That understates the findings, because pass/fail over an unprotected 96-byte frame = P(zero bit errors), while the campaign's net-rate claims assume an outer code. A config at 1% BER shows ~0% raw pass-rate yet is fully recoverable with the modest FEC its net_bps already prices in. So tx_bits/rx_bits were added: the analyzer now computes raw BER per config → feeds the same project_to_cassette table → reports projected net bps + whole-file recoverability. Now the tape test measures the claims, not just unprotected survival.

12 · Simulation vs. reality — the load-bearing caveat

This is the most important section to internalize. The campaign was measured at 42 dB SNR, 0.10% flutter. The real cassette acoustic loop (laptop → tape → deck → speaker → iPhone mic), measured in experiments/dpd/channel_model.json, is far harsher:

~30 dB noisier, ~20× more flutter. OFDM-QAM needs ~13 dB just for QPSK, so the acoustic loop sits right at its cliff. The simulated dry-run through the worn + 0.88× channel (the acoustic-loop proxy) predicts:

The headline reversal. The simulated 3.69× OFDM champion does not transfer to the harsh acoustic loop — coherent QAM is destroyed by 2.2% flutter and deep nulls. The non-coherent combinatorial k-of-M scheme becomes the real-tape winner (c2_m32_k2, ~729 net bps recoverable) precisely because energy-only detection shrugs off the wobble that wrecks phase-tracking demods. To realize the 1.5–4 kbps numbers you need the electrical line-in path (a USB interface like the UCA222) that removes the acoustic hop and most of the flutter — which is exactly why the master tests both paths.

One honesty note: c4_bpsk's BER is measured at its repetition-coded layer, so its projected net is pessimistically low — but it's the weakest OFDM rung regardless, so it doesn't change the picture.

14 · Glossary

  • FSK / MFSK — Frequency-Shift Keying: encode bits by which tone is present. M-ary = M tones, log₂M bits each.
  • Non-coherent detection — decide by energy only (FFT magnitude), ignoring phase. Robust to flutter.
  • Coherent detection — track and use phase (needed by QAM). More efficient, but flutter-fragile.
  • Orthogonality — tones spaced so their FFT bins don't leak into each other (Δf·Tsym = 1).
  • Gray code — an ordering where adjacent values differ in one bit; turns a near-miss into a 1-bit error.
  • OFDM — many narrow subcarriers in parallel via IFFT; each carries a QAM symbol.
  • QAM — Quadrature Amplitude Modulation: bits encoded in a 2-D grid of amplitude×phase points.
  • Cyclic prefix (CP) — a copy of the symbol tail prepended; absorbs timing slop / short echoes.
  • Bit-loading / water-filling — give each subcarrier a QAM order matched to its SNR.
  • Pilot — a known symbol used by the receiver to estimate and undo the channel per subcarrier.
  • EVM — Error Vector Magnitude: how far received constellation points land from ideal (dB; higher = cleaner).
  • PAPR / crest factor — peak-to-average power ratio; high PAPR clips/overdrives the tape → intermodulation.
  • IMD — Intermodulation Distortion: spurious tones from non-linearity (the −10.5 dB floor).
  • LLR — Log-Likelihood Ratio: a soft, signed confidence that a bit is 0 vs 1.
  • Viterbi — optimal decoder for convolutional codes; "soft" Viterbi uses LLRs not hard bits.
  • Reed-Solomon / fountain / MDS — outer codes; fountain (RaptorQ) recovers k symbols from any k/(1−e) received.
  • Erasure vs error — a known-lost slot (cheap to code) vs a silently-flipped bit (expensive).
  • BER — Bit Error Rate. P_full — probability the whole file is recovered.
  • FTN — Faster-than-Nyquist: deliberately overlapping symbols past orthogonality, decoded against the interference.
  • Wow / flutter — slow / fast tape-speed variation warping the time axis (WRMS = weighted RMS %).
